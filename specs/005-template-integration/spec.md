## User Scenarios & Testing *(mandatory)*

### User Story 1 — Default GOST Template Output (Priority: P1)

Как пользователь конвертера Typst→DOCX, я хочу, чтобы по умолчанию сгенерированный документ соответствовал ГОСТ 7.32-2017, используя готовый шаблон ВКР, чтобы не тратить время на ручное форматирование в Word.

**Why this priority**: Это решает ключевую проблему текущей реализации — ручная настройка `_configure_styles()` не работает полностью для Heading стилей (заголовки остаются Calibri). Использование готового шаблона гарантирует корректное форматирование.

**Independent Test**: Можно протестировать конвертацией тестового Typst-файла и проверкой, что сгенерированный DOCX использует стили из шаблона (Times New Roman, чёрные заголовки, отступ первой строки) без вызова `_configure_styles()`.

**Acceptance Scenarios**:

1. **Given** пользователь запускает конвертацию без указания `--reference-doc`, **When** конвертер создаёт DOCX, **Then** документ использует встроенный шаблон ВКР как `reference_doc` и стили применяются корректно
2. **Given** шаблон ВКР содержит стиль Normal с Times New Roman 14pt, **When** конвертер создаёт параграфы, **Then** они используют этот стиль без дополнительной ручной настройки шрифтов
3. **Given** шаблон ВКР содержит Heading 1-3 с чёрным цветом, **When** конвертер создаёт заголовки, **Then** они отображаются чёрным цветом (не синим по умолчанию Word)

---

### User Story 2 — Custom Template Styles for Structural Elements (Priority: P2)

Как пользователь, я хочу, чтобы структурные элементы документа (титульный лист, введение, заключение, подписи рисунков, названия таблиц) использовали специализированные стили из шаблона ВКР для полного соответствия ГОСТ.

**Why this priority**: Шаблон ВКР содержит кастомные стили (`Титул`, `Заг_не_содержание`, `Подпись рисунков`, `Таблица название`, `Формулы`), которые обеспечивают правильное форматирование специфичных элементов академического документа.

**Independent Test**: Можно протестировать отдельно, создав DOCX с элементами, требующими кастомных стилей, и проверив, что применяются правильные стили из шаблона.

**Acceptance Scenarios**:

1. **Given** документ содержит ненумерованный структурный заголовок (например, "ВВЕДЕНИЕ"), **When** конвертер записывает его в DOCX, **Then** используется стиль `Заг_не_содержание` (центрирование, полужирный, разрыв страницы)
2. **Given** документ содержит подпись к рисунку, **When** конвертер записывает её, **Then** используется стиль `Подпись рисунков` (центрирование, без отступа первой строки)
3. **Given** документ содержит название таблицы, **When** конвертер записывает его, **Then** используется стиль `Таблица название` (курсив отключён, 14pt, keepNext)

---

### User Story 3 — CLI Template Path Override (Priority: P3)

Как пользователь, я хочу иметь возможность указать свой шаблон DOCX через CLI или использовать шаблон по умолчанию, чтобы иметь гибкость в выборе оформления.

**Why this priority**: Пользователи могут иметь собственные шаблоны университета или хотеть использовать стандартный Word-шаблон. Это обеспечивает обратную совместимость и гибкость.

**Independent Test**: Можно протестировать запуском CLI с разными значениями `--reference-doc` (путь к файлу, отсутствие опции) и проверкой результата.

**Acceptance Scenarios**:

1. **Given** пользователь указывает `--reference-doc /path/to/custom.docx`, **When** конвертер создаёт DOCX, **Then** используется указанный шаблон вместо встроенного
2. **Given** пользователь не указывает `--reference-doc`, **When** конвертер создаёт DOCX, **Then** автоматически используется встроенный шаблон ВКР
3. **Given** встроенный шаблон отсутствует или повреждён, **When** конвертер инициализируется, **Then** происходит fallback на стандартный пустой Document() с предупреждением

---

### Edge Cases

- Что происходит, если встроенный шаблон ВКР отсутствует в установленном пакете?
- Как система обрабатывает шаблон, в котором отсутствуют ожидаемые стили (например, `Заг_не_содержание`)?
- Что происходит, если пользовательский шаблон содержит стили с теми же именами, но другими характеристиками?
- Как ведёт себя система при конвертации на Windows vs Linux (пути к шаблону)?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА использовать DOCX-шаблон `Шаблон_оформления_ВКР_2026_новый.docx` как `reference_doc` по умолчанию при создании DOCX, если пользователь не указал другой шаблон
- **FR-002**: Система ДОЛЖНА убрать вызов `_configure_styles()` при использовании шаблона, полагаясь на встроенные стили шаблона
- **FR-003**: Система ДОЛЖНА поддерживать fallback на стандартный `Document()` при отсутствии или недоступности шаблона, с выводом предупреждения
- **FR-004**: Система ДОЛЖНА использовать стиль `Заг_не_содержание` для ненумерованных структурных заголовков (РЕФЕРАТ, ВВЕДЕНИЕ, ЗАКЛЮЧЕНИЕ, СПИСОК ЛИТЕРАТУРЫ)
- **FR-005**: Система ДОЛЖНА использовать стиль `Подпись рисунков` для подписей к рисункам и `Таблица название` для названий таблиц
- **FR-006**: Система ДОЛЖНА использовать стиль `Table Grid` для таблиц (границы single, #000000, sz=4)
- **FR-007**: Система ДОЛЖНА использовать стиль `Титул` для элементов титульного листа, если таковой присутствует в документе
- **FR-008**: Система ДОЛЖНА обновить CLI для приёма опции `--reference-doc` с путём к пользовательскому шаблону
- **FR-009**: Система ДОЛЖНА сохранять обратную совместимость — существующие вызовы без шаблона продолжают работать

### Key Entities *(include if feature involves data)*

- **DOCX Template**: Файл `.docx`, содержащий предопределённые стили, настройки страницы и структуру. Ключевые атрибуты: стили абзацев, стили символов, стили таблиц, настройки страницы (поля, размер), колонтитулы.
- **Style Mapping**: Сопоставление между типами элементов IR (Intermediate Representation) и именами стилей в шаблоне. Например: `Section(level=1)` → `"Heading 1"`, `Caption(kind=FIGURE)` → `"Подпись рисунков"`.
- **Reference Document Path**: Путь к файлу шаблона, используемому при инициализации `python-docx Document()`. Может быть встроенным (в пакете) или пользовательским.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Сгенерированный DOCX использует Times New Roman 14pt для основного текста без ручной настройки шрифтов в коде
- **SC-002**: Заголовки первого уровня отображаются чёрным цветом, по центру, с разрывом страницы (как в шаблоне ВКР)
- **SC-003**: Подписи к рисункам центрированы и не имеют отступа первой строки (соответствие ГОСТ 7.32-2017)
- **SC-004**: Таблицы имеют чёрные границы (single, sz=4) в соответствии со стилем `Table Grid`
- **SC-005**: Пользователь может переопределить шаблон через CLI опцию `--reference-doc` и получить документ с пользовательскими стилями
- **SC-006**: При отсутствии шаблона система генерирует валидный DOCX с предупреждением (degradation graceful)

## Assumptions

- Шаблон ВКР (`Шаблон_оформления_ВКР_2026_новый.docx`) содержит корректные стили ГОСТ 7.32-2017 и будет включён в дистрибутив пакета
- `python-docx` корректно загружает и применяет стили из `reference_doc` при создании `Document(reference_doc)`
- Пользователи предпочитают ГОСТ-совместимый вывод по умолчанию для академических документов
- Кастомные стили шаблона (`Заг_не_содержание`, `Подпись рисунков`, `Таблица название`) присутствуют в шаблоне; если отсутствуют — используется fallback на стандартные стили Word
- Изменения не затрагивают логику парсинга Typst (Ingest → Parser → IR слои остаются неизменными)
