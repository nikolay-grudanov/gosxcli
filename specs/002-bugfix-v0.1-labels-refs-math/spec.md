## User Scenarios & Testing

### User Story 1 — Извлечение ссылок (P0)

**Описание**: Пользователь добавляет в Typst-документ ссылки вида `@fig:results`, `@tbl-data`, `@eq:main`. При конвертации в DOCX ссылки должны правильно разрешаться и отображаться.

**Приоритет**: P0 (Critical)

**Почему это важно**: Ссылки — ключевая функция академических документов. Без них документ теряет смысл.

**Независимое тестирование**: Можно протестировать отдельно через `test_extractor_v2_multiple_references` и `test_ref_extraction`.

**Acceptance Scenarios**:

1. **Given** текст с `@fig:1, @tbl-2, @eq-3`, **When** запускается парсер, **Then** извлекается 3 CrossReference с правильными target_label.
2. **Given** параграф с текстом "Как видно на @fig:results", **When** запускается парсер, **Then** параграф содержит и текст, и CrossReference.
3. **Given** пустой документ без ссылок, **When** запускается парсер, **Then** ошибок не возникает, CrossReferences не создаются.

---

### User Story 2 — Извлечение меток (P0)

**Описание**: Пользователь добавляет метки `<fig:results>`, `<table:data>` к элементам документа. При конвертации метки должны связываться с соответствующими элементами.

**Приоритет**: P0 (Critical)

**Почему это важно**: Метки — основа для работы ссылок. Без меток ссылки не имеют цели.

**Независимое тестирование**: Можно протестировать через `test_label_extraction` и `test_extractor_v2_table`.

**Acceptance Scenarios**:

1. **Given** figure с меткой `<fig:test>`, **When** запускается парсер, **Then** Figure.label == "fig:test".
2. **Given** таблица с меткой `<tbl:data>`, **When** запускается парсер, **Then** Table.label == "tbl:data".
3. **Given** несколько меток в документе, **When** запускается парсер, **Then** все метки извлекаются и не дублируются.
4. **Given** документ без меток, **When** запускается парсер, **Then** все элементы имеют label == None.

---

### User Story 3 — Парсинг математических формул (P1)

**Описание**: Пользователь добавляет математические формулы с греческими буквами (`\alpha`, `\beta`, `\gamma`). При конвертации формулы должны сохранять греческие символы.

**Приоритет**: P1 (High)

**Почему это важно**: Греческие буквы — стандарт научных формул. Без них невозможно корректно отобразить большинство уравнений.

**Независимое тестирование**: Тестируется через `test_math_with_greek`.

**Acceptance Scenarios**:

1. **Given** формула `$$\alpha + \beta = \gamma$$`, **When** запускается парсер, **Then** Equation.latex содержит `\alpha`, `\beta`, `\gamma`.
2. **Given** формула `$$\sum_{i=1}^{n} x_i$$`, **When** запускается парсер, **Then** Equation.latex сохраняет индексы.
3. **Given** пустая формула `$$$$`, **When** запускается парсер, **Then** Equation создаётся с пустым latex.

---

### User Story 4 — Извлечение таблиц (P1)

**Описание**: Пользователь создаёт таблицы с помощью `#table(...)`. При конвертации таблицы должны извлекаться с правильной структурой строк и ячеек.

**Приритет**: P1 (High)

**Почему это важно**: Таблицы — ключевой элемент академических документов для представления данных.

**Независимое тестирование**: Тестируется через `test_literature_review_tables`.

**Acceptance Scenarios**:

1. **Given** простая таблица с 2 колонками и 2 строками, **When** запускается парсер, **Then** извлекается 1 Table с 2 строками.
2. **Given** таблица с `table.header()`, **When** запускается парсер, **Then** Table.has_header == True.
3. **Given** пустая таблица `#table(columns: 1)`, **When** запускается парсер, **Then** Table создаётся, строк может быть 0.

---

### Edge Cases

- Ссылка без цели (`@` без label) — как система должна обрабатывать?
- Метка без элемента (orphan label `<fig:orphan>`) — привязывать к ближайшему элементу или оставлять несвязанной?
- Дублирующиеся метки — какая метка "выигрывает"?
- Ссылка на несуществующую метку — Warning или ошибка в строгом режиме?
- Формула с неизвестным LaTeX-символом — оставить как есть или заменить на placeholder?
- Пустая строка в таблице — пропускать или создавать пустую ячейку?
- Unicode-символы в формулах — поддержка кириллицы в индексах?

## Требования

### Functional Requirements

#### FR-001: Исправление токена REF

- **FR-001.1**: Система ДОЛЖНА распознавать токен `REF_INLINE` из сканера вместо несуществующего `REF`
- **FR-001.2**: Тип токена в extractor_v2.py ДОЛЖЕН соответствовать типу, генерируемому scanner.py
- **FR-001.3**: В методе `_extract_paragraph_content` необходимо заменить `token.type == "REF"` на `token.type == "REF_INLINE"`

#### FR-002: Правильное извлечение target_label

- **FR-002.1**: При извлечении ссылки ДОЛЖЕН сохраняться полный label включая префикс (`fig:results`, а не `fig`)
- **FR-002.2**: Паттерн для извлечения target_label ДОЛжен использовать `ref_match` напрямую, без дополнительной обработки
- **FR-002.3**: В тестах `test_extractor_v2_multiple_references` ожидаемое значение ДОЛЖНО быть `"@fig-1"`, `"@tbl-2"`, `"@eq-3"` — с @

#### FR-003: Извлечение меток

- **FR-003.1**: Метки ДОЛЖНЫ извлекаться из токена LABEL в scanner.py
- **FR-003.2**: Метка ДОЛЖНА быть привязана к элементу, который идёт после неё (figure, table, equation)
- **FR-003.3**: При наличии метки у элемента label ДОЛЖЕН быть строкой, не None

#### FR-004: Сохранение греческих букв в формулах

- **FR-004.1**: LaTeX с греческими буквами ДОЛЖЕН сохраняться без изменений
- **FR-004.2**: Scanner НЕ ДОЛЖЕН обрабатывать `\` как начало специальной последовательности
- **FR-004.3**: Блок math должен включать весь текст между `$$...$$` включая `\alpha`, `\beta` и т.д.

#### FR-005: Извлечение таблиц

- **FR-005.1**: Парсер ДОЛЖЕН распознавать `#table(` токен
- **FR-005.2**: Содержимое таблицы ДОЛЖНО парситься между открывающей и закрывающей скобками
- **FR-005.3**: Строки таблицы ДОЛЖНЫ извлекаться из содержимого между `[...]` скобками

### Key Entities

- **Token**: Тип из scanner.py (TEXT, HEADING, LABEL, REF_INLINE, BLOCK_MATH_DELIM, TABLE_START и т.д.)
- **CrossReference**: IR-узел с полем target_label
- **Label**: Связывается с Figure, Table, Equation через поле .label
- **Equation**: IR-узел с полем .latex для хранения формулы
