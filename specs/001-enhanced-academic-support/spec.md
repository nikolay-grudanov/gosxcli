## User Scenarios & Testing *(mandatory)*

### User Story 1 - Academic Thesis Conversion (Priority: P1)

Автор академической работы (магистерской диссертации) конвертирует многофайловый Typst документ в формат DOCX для подачи в университет. Документ содержит математические формулы, сложные таблицы с объединёнными ячейками, перекрёстные ссылки и библиографические ссылки. После конвертации автор получает DOCX файл, который корректно открывается в Microsoft Word или LibreOffice, соответствует стандарту ГОСТ 7.32-2017, содержит все формулы, таблицы, ссылки и оглавление.

**Why this priority**: Это основной целевой кейс проекта v0.2. Без успешной конвертации реального документа проект не переходит в режим практического использования.

**Independent Test**: Можно протестировать на анонимизированной версии документа `00-vvedenie.typ` и `01-literature-review.typ`. Конвертер генерирует DOCX без исключений, файл открывается в Word/LibreOffice без ошибок, структура документа (заголовки, таблицы, формулы) сохранена.

**Acceptance Scenarios**:

1. **Given** автор имеет многофайловый Typst проект с главами, включёнными через `#include`, **When** автор запускает конвертер, **Then** система загружает все файлы рекурсивно (максимум 10 уровней глубины), объединяет их в единый IR и генерирует DOCX без ошибок
2. **Given** документ содержит математические формулы разной сложности (дроби, суммы, интегралы, матрицы, индексы), **When** конвертер обрабатывает формулы, **Then** формулы рендерятся через OMML (Office Math Markup Language) для поддержки редактирования в Word, при невозможности конвертации - рендерятся как изображение с логированием ошибки
3. **Given** документ содержит сложные таблицы с атрибутами `columns`, `stroke`, `fill`, `align`, `colspan`, `rowspan`, **When** конвертер генерирует DOCX, **Then** таблицы имеют правильную ширину столбцов, границы, затенение заголовка, объединённые ячейки
4. **Given** документ использует перекрёстные ссылки `@fig:results`, `@tbl:comparison`, `<ch01>`, **When** конвертер резолвит ссылки, **Then** ссылки отображаются локализованным текстом («Рисунок 1.2», «Таблица 2.1», «Глава 1») с chapter-aware нумерацией

### User Story 2 - Rich Text Formatting Support (Priority: P2)

Автор академической работы использует инлайн-форматирование текста (жирный, курсив, код, ссылки) для выделения терминов, определений и ссылок на источники. После конвертации DOCX файл сохраняет всё инлайн-форматирование.

**Why this priority**: Поддержка инлайн-форматирования критически важна для академических документов, но может быть реализована независимо от сложных таблиц и формул.

**Independent Test**: Можно протестировать на фикстуре `fixtures/inline_formatting.typ`. Конвертер генерирует DOCX, где параграфы содержат mixed runs (bold + normal + code в одном параграфе).

**Acceptance Scenarios**:

1. **Given** параграф содержит `*жирный текст*`, `_курсив_`, `` `код` `` и обычный текст, **When** конвертер обрабатывает параграф, **Then** DOCX содержит соответствующие formatting runs (<w:b/>, <w:i/>, <w:rStyle w:val="Code"/>)
2. **Given** параграф содержит гиперссылку `#link("url")[текст]`, **When** конвертер обрабатывает параграф, **Then** DOCX содержит элемент <w:hyperlink r:id="..."> с внешней ссылкой
3. **Given** фикстуры v0.1 используют параграфы с простым текстом (content: str), **When** конвертер обрабатывает такие параграфы, **Then** поддерживается обратная совместимость через fallback на content, когда runs пуст

### User Story 3 - Reference Validation and Error Reporting (Priority: P2)

Автор академической работы использует множество перекрёстных ссылок на рисунки, таблицы, формулы и главы. После конвертации система сообщает о всех неразрешённых ссылках и не используемых метках, помогая автору исправить документ.

**Why this priority**: Помощь автору в выявлении ошибок в документе критически важна для качества результата.

**Independent Test**: Можно протестировать на документе с намеренно добавленными неразрешёнными ссылками. Конвертер выводит WARNING для каждой undefined reference и INFO для каждой unreferenced label.

**Acceptance Scenarios**:

1. **Given** документ содержит определённые метки `<fig:results>`, `<tbl:comparison>` и ссылку на несуществующую метку `@fig:missing`, **When** конвертер завершает обход IR, **Then** система выводит WARNING для `@fig:missing` и INFO для меток, которые определены но не используются
2. **Given** конвертер запущен с флагом `--strict` и документ содержит undefined reference, **When** конвертер завершает работу, **Then** процесс завершается с EXIT_CODE=1
3. **Given** документ содержит 3 главы с рисунками, **When** конвертер нумерует рисунки, **Then** рисунки нумеруются в формате «Рисунок X.Y» (глава.номер_в_главе), счётчики сбрасываются при переходе к новой главе

### User Story 4 - Performance Benchmarking and Regression Testing (Priority: P3)

Разработчик системы внедряет регрессионное тестирование и бенчмаркинг для обеспечения стабильности качества конвертации при развитии проекта. При изменении кода Writer автоматически сравниваются результаты с эталонными файлами, измеряется время конвертации на документах разного размера.

**Why this priority**: Обеспечение качества и производительности важно для развития проекта, но не блокирует базовую функциональность.

**Independent Test**: Можно запустить `make regression` и `make benchmark`. Система генерирует отчёты о расхождениях с эталонными файлами и замеряет время конвертации.

**Acceptance Scenarios**:

1. **Given** существует эталонный DOCX файл для фикстуры `fixtures/minimal.typ`, **When** разработчик вносит изменения в Writer, **Then** система автоматически сравнивает сгенерированный DOCX с эталоном и сообщает о расхождениях
2. **Given** конвертер запущен с флагом `--benchmark`, **When** конвертер обрабатывает `minimal.typ` и `01-literature-review.typ`, **Then** результаты записываются в `benchmarks/results.json`, время конвертации `minimal.typ` < 1 сек, `01-literature-review.typ` < 10 сек
3. **Given** E2E тесты запущены на реальных документах (`00-vvedenie.typ`, `01-literature-review.typ`), **When** тесты завершены, **Then** проверяется структура DOCX (количество заголовков, наличие таблиц, отсутствие пустых параграфов)

### Edge Cases

- Что происходит при циклических `#include` ссылках (file A includes B, B includes A)? Система должна ограничивать глубину рекурсии до 10 уровней и генерировать ошибку при превышении
- Как система обрабатывает отсутствующий include-файл? Система должна генерировать ERROR и прерывать конвертацию
- Что происходит при невозможности конвертировать математическое выражение в OMML? Система должна рендерить выражение как изображение (fallback) и логировать ошибку с указанием строки и типа ошибки
- Как система обрабатывает таблицы без заголовков (отсутствие `table.header([...])`)? Система должна генерировать таблицу без выделения заголовка
- Что происходит при ссылке на несуществующую метку в `--strict` режиме? Система должна завершать с EXIT_CODE=1
- Как система обрабатывает пустые параграфы? Система должна минимизировать количество пустых параграфов в сгенерированном DOCX
- Что происходит при несоответствии количества ячеек в строке таблицы количеству столбцов? Система должна генерировать WARNING и заполнять недостающие ячейки пустым содержимым

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST поддерживать рекурсивную загрузку Typst файлов через директиву `#include "path/to/file.typ"` с ограничением глубины 10 уровней для защиты от циклических ссылок
- **FR-002**: System MUST обрабатывать относительные пути в `#include` относительно текущего файла и поддерживать структуру многофайлового проекта
- **FR-003**: System MUST логировать дерево загруженных файлов при запуске с флагом `--debug` для диагностики
- **FR-004**: System MUST поддерживать рендеринг математических формул через Office Math Markup Language (OMML) для основных типов выражений: дроби (\frac{}{}), суммы и интегралы (\sum, \int с пределами), матрицы (\begin{matrix}...\end{matrix}), индексы и степени (x^2, x_i, x_i^j), греческие символы (\alpha, \beta, \gamma)
- **FR-005**: System MUST использовать `latex2mathml` как обязательную зависимость (core dependency) для конвертации LaTeX выражений в MathML/OMML
- **FR-006**: System MUST рендерить математические выражения как изображения (fallback) при невозможности конвертировать в OMML, вместо текстового placeholder
- **FR-007**: System MUST логировать каждое выражение, которое ушло в image-fallback, с указанием строки и типа ошибки
- **FR-008**: System MUST поддерживать флаг `--math-mode [native|image|fallback]` для выбора режима рендеринга математических выражений
- **FR-009**: System MUST поддерживать в IR модель MathNode с полями `render_mode: str` и `render_error: Optional[str]`
- **FR-010**: System MUST распознавать атрибуты таблицы `columns` с процентными ширинами и явными спецификациями выравнивания, сохраняя как список `ColSpec(width: str, align: str)`
- **FR-011**: System MUST парсить атрибут `stroke` (например, `0.7pt`) и сохранять в IR как `border_width: float`
- **FR-012**: System MUST распознавать паттерн «header row highlight» в лямбда-выражениях `fill` (например, `(col, row) => if row == 0 { luma(220) }`) и устанавливать флаг `header_shaded: bool = True`
- **FR-013**: System MUST распознавать паттерн «header center, body left» в лямбда-выражениях `align` и сохранять `header_align: str`, `body_align: str`
- **FR-014**: System MUST выделять `table.header([...])` в отдельный `TableHeaderNode` в IR
- **FR-015**: System MUST поддерживать парсинг `table.cell(colspan: N)[...]` и `table.cell(rowspan: N)[...]` с сохранением в IR как `TableCellNode(colspan: int = 1, rowspan: int = 1)`
- **FR-016**: System MUST генерировать корректный OpenXML `<w:gridSpan>` и `<w:vMerge>` для объединённых ячеек в таблицах
- **FR-017**: System MUST применять атрибуты таблиц к DOCX: `border_width` к `<w:tcBorders>`, затенение заголовка (`w:shd`) при `header_shaded = True`, выравнивание ячеек по `header_align` / `body_align`, ширины столбцов через `<w:tblGrid>` / `<w:gridCol w:w="...">`
- **FR-018**: System MUST распознавать вложенные таблицы `table(... table(...) ...)` как `NestedTableNode` и генерировать `<w:tbl>` внутри `<w:tc>`
- **FR-019**: System MUST поддерживать chapter-aware нумерацию рисунков, таблиц и формул в формате «Рисунок X.Y» (глава.номер_в_главе)
- **FR-020**: System MUST отслеживать текущую главу в контексте Writer через `ChapterContext(chapter: int, fig_counter: int, tbl_counter: int, eq_counter: int)`
- **FR-021**: System MUST сбрасывать счётчики (fig_counter, tbl_counter, eq_counter) при переходе к новой главе уровня 1
- **FR-022**: System MUST форматировать перекрёстные ссылки локализованным текстом: `@fig:results` → «Рисунок 1.2», `@tbl:abm-comparison` → «Таблица 2.1», `@eq:formula` → «Формула (1.3)», `<ch01>` → «Глава 1»
- **FR-023**: System MUST определять паттерн ссылки по prefix-части метки: `fig:`, `tbl:`, `eq:`, `ch:`
- **FR-024**: System MUST поддерживать конфигурацию локализации в `config.py`: `ref_labels = {"fig": "Рисунок", "tbl": "Таблица", "eq": "Формула", "ch": "Глава"}`
- **FR-025**: System MUST выполнять bidirectional validation после полного обхода IR: сравнивать множество defined_labels vs referenced_labels
- **FR-026**: System MUST выводить WARNING для каждой undefined reference и INFO для каждой unreferenced label (defined but never used)
- **FR-027**: System MUST завершаться с EXIT_CODE=1 при любой undefined reference в `--strict` режиме
- **FR-028**: System MUST поддерживать конвертацию `#outline()` в оглавление через `python-docx` Table of Contents field (`w:fldChar`)
- **FR-029**: System MUST поддерживать инлайн-форматирование: `*text*` → `<w:b/>` (bold) в `InlineRunNode(bold=True)`, `_text_` → `<w:i/>` (italic) в `InlineRunNode(italic=True)`
- **FR-030**: System MUST поддерживать инлайн-код: `` `code` `` → `<w:rStyle w:val="Code"/>` в `InlineCodeNode`
- **FR-031**: System MUST поддерживать гиперссылки: `#link("url")[text]` → `<w:hyperlink r:id="...">` с внешней ссылкой
- **FR-032**: System MUST заменять `ParagraphNode.content: str` на `ParagraphNode.runs: List[InlineNode]` для поддержки mixed formatting в одном параграфе
- **FR-033**: System MUST поддерживать обратную совместимость с v0.1 фикстурами: если `runs` пуст, фолбэк на `content`
- **FR-034**: System MUST поддерживать E2E тестирование на реальных документах: `fixtures/real_vkr/` с анонимизированной версией `00-vvedenie.typ` и `01-literature-review.typ`
- **FR-035**: System MUST проверять структуру сгенерированного DOCX: количество заголовков, наличие таблиц, отсутствие пустых параграфов
- **FR-036**: System MUST поддерживать regression suite: golden file тесты для ключевых фикстур, автоматическое сравнение с эталоном при изменении Writer
- **FR-037**: System MUST поддерживать performance benchmark: замер времени конвертации на `minimal.typ` (< 1 сек) и на `01-literature-review.typ` (< 10 сек), запись результатов в `benchmarks/results.json`
- **FR-038**: System MUST поддерживать флаг `--benchmark` в CLI для запуска бенчмаркинга

### Key Entities *(include if feature involves data)*

- **ChapterContext**: Контекст текущей главы для chapter-aware нумерации ссылок. Содержит номер главы (chapter), счётчики для различных типов элементов (fig_counter, tbl_counter, eq_counter). Используется Writer для генерации локализованных ссылок.
- **ColSpec**: Спецификация столбца таблицы. Содержит ширину столбца (width: str, например "17%") и выравнивание (align: str). Используется для генерации корректной структуры таблиц в DOCX.
- **InlineNode**: Базовый тип для инлайн-элементов параграфа. Подтипы включают InlineRunNode (текст с форматированием bold/italic), InlineCodeNode (код), CrossRefNode (перекрёстная ссылка).
- **MathNode**: Узел математического выражения. Содержит LaTeX выражение (expression: str), режим рендеринга (render_mode: str: native|image|fallback) и ошибку рендеринга (render_error: Optional[str]).
- **TableHeaderNode**: Узел заголовка таблицы. Выделяется из общей структуры таблицы для отдельной обработки стилей (затенение, выравнивание).
- **TableCellNode**: Узел ячейки таблицы. Содержит содержимое ячейки (content: str или runs: List[InlineNode]), объединение ячеек (colspan: int = 1, rowspan: int = 1).
- **TableNode**: Узел таблицы. Содержит список спецификаций столбцов (columns: List[ColSpec]), ширину границ (border_width: float), флаг затенения заголовка (header_shaded: bool), выравнивание заголовка и тела (header_align: str, body_align: str), список ячеек (cells: List[List[TableCellNode]]).

## Assumptions

- Пользователь имеет установленный Python 3.12+ и все необходимые зависимости, включая `latex2mathml` (теперь core dependency)
- Пользователь использует Typst 0.11+ с поддержкой пакетов, в том числе `@preview/modern-g7-32:0.2.0`
- Microsoft Word и LibreOffice поддерживают Office Math Markup Language (OMML) для редактирования формул
- Структура реального документа `thesis.typ` соответствует описанию в спецификации: использует `#include` для глав, содержит сложные таблицы, формулы, библиографические ссылки
- Зависимость `latex2mathml` поддерживает конвертацию основных LaTeX конструкций в MathML/OMML
- Пользователь понимает концепцию chapter-aware нумерации и использует метки с соответствующими префиксами (`fig:`, `tbl:`, `eq:`, `ch:`)
- DOCX файл, сгенерированный через `python-docx`, совместим с Microsoft Word 2010+ и LibreOffice 6.0+
- Ограничение глубины рекурсии `#include` до 10 уровней достаточно для реальных проектов и защищает от циклических ссылок
- Формулы, которые невозможно конвертировать в OMML, могут быть рендерены как изображения с приемлемым качеством
- Поддержка `#outline()` для генерации оглавления в DOCX ограничена базовой функциональностью (без кастомизации стилей)
